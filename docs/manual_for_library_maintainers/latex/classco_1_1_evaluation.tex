\doxysubsection{Detailed Description}
\subsubsection*{template$<$class T, Config\+Computation M, Config\+Output O$>$\newline
class co\+::\+Evaluation$<$ T, M, O $>$}

Abstract evaluation class. This class cannot be used directly but serves as a parent for problem specific subclasses like \mbox{\hyperlink{classco_1_1_biogas_evaluation}{Biogas\+Evaluation}}. It is templatized by the numeric datatype (like EFloat64) used by the problem and the way the objective function f(x) is evaluated by the system. Before creating your own \mbox{\hyperlink{classco_1_1_evaluation}{Evaluation}} child class, it should always be investigated if existing child classes like \mbox{\hyperlink{classco_1_1_biogas_evaluation}{Biogas\+Evaluation}} represent the problem well enough. When an existing evaluation child class does not represent a problem well, a new class inheriting from \mbox{\hyperlink{classco_1_1_evaluation}{Evaluation}} should be created.

Abstract class that represents how problem data is loaded (like experimental data etc.). The data is saved in two containers. The first represent time data (the t values in a problem of the form f(t)) and the second codomain data (the function values resulting from f(t)). If the problem also includes spatial domain data x (like f(t,x)) a new function should probably be created that also saves this spatial data. The template parameter T represents the numeric datatype (such as EFloat64) underlying the problem.
